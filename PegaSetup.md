# Pega Integration Guide

This guide describes how to connect a **Pega Platform** application to:
- **Azure Event Hubs** (Kafka endpoint) for message produce/consume.
- **Apicurio Schema Registry** (Confluent-compatible API) for JSON Schema support.

The configuration is delivered via a single, self-contained `client.properties` file.

---

## Prerequisites

- **Pega Platform 8.7 or later**.
- The `client.properties` file obtained from **Actions ‚Üí Download client.properties**.

> **Important:** Do not edit this file. It is generated by Terraform and contains the correct
> credentials and endpoints for this environment. If anything is wrong, the fix must be applied to the infrastructure (Terraform), not the file.

---

## Step 1 ‚Äî Retrieve the properties file

1. Go to **Actions ‚Üí Download client.properties** in the GitHub repository.
2. Download the `client-properties` artifact.
3. Extract `client.properties`.

---

## Step 2 ‚Äî Configure Kafka in Pega

### 2a ‚Äî Create Kafka Configuration

In **Pega Designer Studio**:

1. Navigate to **Records ‚Üí Integration-Resources ‚Üí Kafka Configuration**.
2. Create a new rule (e.g., `EventHubKafkaConfig`).
3. Set **Bootstrap servers** to the value of `bootstrap.servers` from `client.properties`.
4. Under **Additional client properties**, paste the contents of `client.properties`.

### 2b ‚Äî JSON Schema Support

The `client.properties` file contains all necessary settings for the Confluent Schema Registry client, including:
- `schema.registry.url`
- `bearer.auth.*` credentials for OAuth2 authentication.

These properties are consumed natively by the Confluent Serdes. The registry acquires the OAuth token automatically using the provided client credentials.

---

## Step 3 ‚Äî Create a Kafka Data Set Rule

1. Navigate to **Records ‚Üí Integration-Resources ‚Üí Data Set**.
2. Create a **Kafka** data set.
3. Set **Kafka Configuration** to `EventHubKafkaConfig`.
4. Set **Topic** to your target topic.
5. Set **Message format** to **JSON**. If using JSON Schema validation, ensure the model matches the registered schema.

---

## Step 4 ‚Äî Verify connectivity

Produce a test message and verify Pega can consume it.

Example payload:
```json
{"check": "pega-test", "ts": "2026-04-26T12:00:00Z"}
```

Verify in Pega:
- The record appears in Data Set browse/run.
- No Kafka consumer or authentication errors appear in the logs.

---

## Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| `SASL authentication failed` | Expired/Wrong credentials | Re-run "Download client.properties" to get fresh credentials. |
| `User is not authenticated` (SR) | OAuth scope mismatch | Verify `bearer.auth.scope` in Terraform matches the Apicurio app registration. |
| `Schema not found` | Schema not registered | Ensure the schema is registered in Apicurio under `<topic>-value`. |
| TLS/SNI error | Wrong bootstrap server | Ensure the bootstrap server matches the custom domain configured in Application Gateway. |
