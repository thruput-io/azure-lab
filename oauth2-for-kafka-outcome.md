## What was built

A production-grade PoC infrastructure that allows external clients (starting with Pega Cloud)
to connect to Azure Event Hubs using standard Confluent Kafka client libraries with Avro schema
support via Apicurio Schema Registry.

The end product for external clients is two self-contained properties files delivered via
GitHub Actions ("Download client.properties" workflow):

| File                       | Purpose                                                                 |
|----------------------------|-------------------------------------------------------------------------|
| `kafka-client.properties`  | Kafka broker connectivity (SASL/PLAIN + `$ConnectionString`)            |
| `schema-client.properties` | Kafka broker + Schema Registry URL + OAuth credentials (self-contained) |

These files are **Terraform outputs** — generated by `entra_app.tf`, stored in Key Vault,
delivered at runtime. They never exist in the repository.

## Architecture

```
External client (Pega)
        │
        ▼ HTTPS :443
Application Gateway (TLS termination, eventhub.grayskull.se, managed-cert)
        │
        ├─► HTTP :8080 → Apicurio Registry (ACI, snet-aci)
        │                  ├── apicurio-registry:3.2.2 (port 8080)
        │                  └── postgres:16-alpine (port 5432, sidecar, ephemeral)
        │                  Auth: Entra ID OIDC (QUARKUS_OIDC_*)
        │                  API:  /apis/ccompat/v7  (Confluent-compatible)
        │
        └─► L4 passthrough :9093 → Azure Event Hubs (Kafka)
                                    Auth: SASL/PLAIN + $ConnectionString
```

**Key Vault** stores:

- `kafka-client-properties` — rendered by `entra_app.tf`
- `schema-client-properties` — rendered by `entra_app.tf`
- `kafka-client-secret` — Entra ID app secret

**GitHub Actions** delivers both properties files as artifacts on demand.

## `schema-client.properties` content (structure)

```properties
# Kafka broker
bootstrap.servers=eventhub.grayskull.se:9093
security.protocol=SASL_SSL
sasl.mechanism=PLAIN
sasl.jaas.config=org.apache.kafka.common.security.plain.PlainLoginModule required \
  username="$ConnectionString" password="<connection-string>";

# Schema Registry (Confluent-compatible API via App Gateway)
schema.registry.url=https://eventhub.grayskull.se/apis/ccompat/v7

# Schema Registry OAuth (Confluent SR client bearer.auth.* properties)
bearer.auth.issuer.endpoint.url=https://login.microsoftonline.com/<tenant_id>/oauth2/v2.0/token
bearer.auth.client.id=<kafka_client_app_id>
bearer.auth.client.secret=<secret>
bearer.auth.scope=api://<apicurio_client_id>/.default
```

The `bearer.auth.*` keys are consumed natively by the Confluent SR client library
(`kafka-avro-serializer` / `kafka-avro-deserializer`). No script-level token handling needed.

## How `schema-check.sh` works

The script (`scripts/schema-check.sh`) is the PoC validation tool. It:

1. **Step 0** — Pre-registers the Avro schema in Apicurio via `curl` (idempotent setup step,
   not an auth bypass). Acquires a token using `bearer.auth.*` from the properties file.
2. **Step 1** — Produces an Avro message using `kafka-avro-console-producer` with
   `--command-config` (Kafka auth) and `--reader-property bearer.auth.*` (SR OAuth).
3. **Step 2** — Consumes the message using `kafka-avro-console-consumer` with
   `--command-config` and `--formatter-property bearer.auth.*`.

The SR client acquires OAuth tokens natively — the script never holds or injects a token
into the SR client. The properties file is the single source of truth.

**Note on `--property value.schema`:** `kafka-avro-console-producer` requires the schema
inline because it has no application code to derive it from. This is a console tool
limitation. Pega's `kafka-avro-serializer` derives the schema from their application code
and registers/looks it up in Apicurio automatically — no inline schema needed.

## Key findings and decisions

### 1. Azure Event Hubs Kafka auth: SASL/PLAIN only (DEV-001)

**Finding:** Azure Event Hubs port 9093 supports **only SASL/PLAIN with `$ConnectionString`**
for Kafka clients. SASL/OAUTHBEARER is not supported.

**Test evidence (2026-04-24):** OAuth token acquired successfully from Entra ID, but the
broker rejected the SASL exchange with `authBytes was serialized as null`. The original
`confluent-endpoint-check.yml` (commit `1561196`) used OAUTHBEARER but was never actually
working at the broker auth level.

**Impact on Pega:** Pega must use SASL/PLAIN + `$ConnectionString` for Kafka broker auth.
OAuth is only available via AMQP (port 5671), which Confluent KafkaSerdes does not use.
This is the documented Azure Event Hubs + Confluent integration pattern — not a workaround.

### 2. Apicurio Schema Registry: Entra ID OAuth via `bearer.auth.*`

Apicurio is configured with Entra ID OIDC (`QUARKUS_OIDC_*`). The Confluent SR client
authenticates using `bearer.auth.*` properties (client credentials flow). The `identifier_uri`
(`api://<apicurio_client_id>`) must be stable — managed via `azuread_application_identifier_uri`
with `lifecycle { ignore_changes = [identifier_uris] }` on the app registration to prevent
Terraform from fighting itself on every deploy.

### 3. `schema-check.sh` evolution

The script went through several iterations before reaching the correct pattern:

| Version                                      | Problem                                                                                    |
|----------------------------------------------|--------------------------------------------------------------------------------------------|
| Original (`confluent-endpoint-check.yml`)    | SASL/OAUTHBEARER for Kafka — never worked                                                  |
| Inline `curl` + `STATIC_TOKEN`               | Worked but violated guidelines — SR client never read `bearer.auth.*` from file            |
| `--reader-property` / `--formatter-property` | **Current** — SR client reads OAuth natively; properties file is the actual source of auth |

The key insight: when `STATIC_TOKEN` was injected, `schema-client.properties` was mounted
but its `bearer.auth.*` keys were consumed by the script's `curl` block — not by the SR
client. The file was effectively ignored for auth. The test passed but proved nothing about
whether the properties file worked as a self-contained config.

### 4. Confluent KafkaSerdes compatibility is the gate

Any feature, workaround, or auth flow that cannot be expressed as standard properties
consumed by `kafka-avro-serializer` / `kafka-avro-deserializer` has no value for Pega.
If a workaround cannot be expressed as standard serdes properties → fix the infrastructure.
Never script around it.

## Guidelines for future AI agents

> **Read `.ai/project_guidelines.md` before touching anything.**

The single most important rule: `kafka-client.properties` and `schema-client.properties`
are **Terraform outputs**. They do not exist in the repository. If a key is missing or
wrong → fix `entra_app.tf` and redeploy. Never create a local file.

The failure mode that has burned multiple AI sessions: a local `.properties` file exists →
AI treats it as authoritative → AI "fixes" the local file → Terraform is never updated →
external clients receive a broken properties file.

See also: `schema-check-cleanup.md` for the detailed rationale behind the current
`schema-check.sh` implementation.
